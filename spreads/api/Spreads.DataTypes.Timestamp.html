Remarks

Timestamp is used as a much faster and convenient replacement of DateTime in Spreads, because DateTime is very problematic data structure in .NET.

First, it has DateTimeKind property that has, in our opinion, meaningless and unusable value Local. When a value of this kind is serialized it doesn't have any meaning outside the serializer machine. Utc and Unspecified have different binary layout, but there is probably no meaningful way to treat unspecified kind other than UTC. The kind bits in DateTime add no value but make the same values binary incompatible. Timestamp time is always UTC and time zone information is supposed to be stored separately.

Second, DateTime structure has Auto, even though internally it is just UInt64 and millions of code lines depend on its binary layout so that Microsoft is unlikely to change it ever. The auto layout makes DateTime and every other structure that contains it as a field not blittable. Timestamp is blittable and very fast. See our serialization documentation on why it is a big deal. (TODO link).

Timestamp is serialized to JSON as decimal string with seconds, e.g. "1552315408.792075401". This is one of the formats supported by microtime package. A value is stored as string to preserve precision, but in JavaScript it could be trivially converted to a number with plus sign +&quot;123456789.123456789&quot;. If you need to keep nanoseconds precision in JavaScript then split the string by dot and use two values separately. Deserializer supports decimal as a number as well as a string.

TimeService provides unique monotonic timestamps and could work across processes via shared memory.

Range: ``` 2^63: 9,223,372,036,854,780,000 Nanos per day: 86,400,000,000,000 (2^47) Nanos per year: 31,557,600,000,000,000 (2^55) 292 years of nanos in 2^63 is ought to be enough for everyone living now and their grand grand grandchildren. ```
